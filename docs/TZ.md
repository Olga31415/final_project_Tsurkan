# Техническое задание: сервис мониторинга качества ML-модели

## 1. Цель

Разработать **сервис мониторинга качества** работающей в продакшене ML-модели: приём потока предсказаний и фактических ответов, расчёт метрик качества в **скользящем временном окне**, сохранение истории для анализа деградации и **генерация алертов** при падении показателей ниже заданных порогов с **гарантированной доставкой** уведомлений (повторные попытки, outbox).

## 2. Контекст и сценарии использования

1. Внешняя система после вызова модели отправляет в сервис запись о предсказании (идентификатор, модель, метка, время, задержка и пр.).
2. Когда становится известен фактический ответ (ground truth), система отправляет его, ссылаясь на тот же `prediction_id`.
3. Оператор или дашборд запрашивает историю метрик по модели и список алертов.
4. При ухудшении `accuracy` или `f1` ниже порогов сервис создаёт алерт и доставляет его во внешний канал (HTTP webhook) или в лог, если URL не задан.

## 3. Функциональные требования

### 3.1. Приём данных

- REST API для приёма **предсказаний** и **фактических ответов** в формате JSON.
- Валидация входных данных (обязательные поля, типы, временные метки с часовым поясом или UTC).
- Идемпотентное обновление предсказаний и сопоставление с фактом по `prediction_id`.

### 3.2. Хранение

- Персистентное хранилище на **SQLite** с режимом **WAL** для устойчивости к конкурентным чтениям/записям в рамках одного хоста.
- Таблицы для предсказаний, фактов, истории метрик, очереди алертов (outbox).

### 3.3. Метрики

В скользящем окне по времени (настраиваемая длительность) для каждой модели рассчитываются:

- `accuracy`, `precision`, `recall`, `f1` (бинарная/категориальная сводка по парам `y_pred` / `y_true`);
- квантили задержки: `latency_p50_ms`, `latency_p95_ms`, `latency_p99_ms` (настраиваемый набор квантилей через окружение).

Результаты агрегации сохраняются в **историю метрик** для построения графиков во времени.

### 3.4. Алерты

- При значении `accuracy` или `f1` **ниже** заданных порогов — создание записи алерта.
- Дедупликация по ключу (модель + метрика + окно/версия логики — в реализации через `dedup_key` в outbox).
- Доставка через таблицу **outbox** с полями статуса, числа попыток, последней ошибки.
- **Exponential backoff** между попытками; максимальное число попыток задаётся конфигурацией.
- При отсутствии `MLQM_WEBHOOK_URL` — запись алерта в **лог** (режим разработки/демо).

### 3.5. Фоновая обработка

- Периодический пересчёт метрик (агрегатор).
- Периодическая отправка due-записей из outbox (диспетчер алертов).
- Ручной endpoint принудительной агрегации для тестов и демонстраций.

### 3.6. Наблюдаемость и эксплуатация

- Endpoint проверки живости (`/health`).
- OpenAPI/Swagger для ручной отладки.
- Настройка через **переменные окружения** (путь к БД, окно, интервалы, пороги, webhook, таймауты).

## 4. Нефункциональные требования

- Язык реализации: **Python 3.11+**.
- Веб-фреймворк: **FastAPI**, ASGI-сервер **Uvicorn**.
- Зависимости зафиксированы в `requirements.txt` и `pyproject.toml`.
- Покрытие ключевой логики **автотестами** (`pytest`): метрики, агрегация с записью в БД и постановкой алертов, повторная доставка алертов, smoke-проверки HTTP API.

## 5. Критерии приёмки

1. Сервис запускается одной командой `python -m ml_quality_monitor` после установки зависимостей.
2. Документация описывает установку, запуск, примеры `curl` и демо-скрипт.
3. `pytest` из **корня репозитория** (с учётом `pytest.ini` и каталога `tests/`) завершается без ошибок в чистом окружении; на GitHub то же проверяет CI.
4. Данные после рестарта сохраняются в файле БД (при неизменном `MLQM_DB_PATH`).

## 6. Ограничения и допущения текущей версии

- Один процесс, фоновые задачи — потоки внутри приложения (не распределённый планировщик).
- Масштабирование по нагрузке ограничено возможностями SQLite; для высоких RPS предполагается смена СУБД и брокера без изменения внешнего контракта API (в README указаны направления эволюции).
